Entrées/Sorties asynchrones pour JavaScript V8.

Un exemple de serveur web écrit en Node qui répond "Bonjour tout le monde" à toutes les requêtes. 

var http = require('http');
http.createServer(function (req, res) {
  res.writeHead(200, {'Content-Type': 'text/plain'});
  res.end('Bonjour tout le monde\n');
}).listen(8124, "127.0.0.1");
console.log('Serveur en place http://127.0.0.1:8124/');

Pour lancer le serveur, écrivez le code dans un fichier exemple.js et éxecutez-le avec le programme node : 

% node exemple.js
Serveur en place http://127.0.0.1:8124/

Voici un exemple de serveur TCP simpliste qui écoute sur le port 8124 et renvoie tout ce qu'il reçoit : 

var net = require('net');

var server = net.createServer(function (socket) {
  socket.write("Serveur echo\r\n");
  socket.pipe(socket);
})

server.listen(8124, "127.0.0.1");

Voir la documentation de l'API pour plus d'exemples.

À propos

Le but de Node est de créer un moyen simple de contruire des programmes distribués et scalables. Dans l'exemple de serveur web ci-dessus, de nombreuses connexions client peuvent être traîtées en concurrence. Node communique avec le système (par epoll, kqueue, /dev/poll/ ou select) pour être notifié lors de nouvelles connexions puis devient inactif. Si quelqu'un d'autre se connecte, il éxecute la fonction de rappel. Chaque connexion ne demande que très peu d'allocation mémoire.

Tout ceci contraste avec le modèle plus commun utilisant des threads. Les application réseau basées sur les threads sont relativement peu performantes et difficile à concevoir. voir : ceci et cela. (anglais) Node utilisera la mémoire beaucoup plus efficacement lors de montées en charge que les systèmes allouant des threads de 2 Mo pour chaque connexion. De plus, les utilisateurs de Node n'ont pas à se préoccuper de verrous mortels. Il n'y a pas de verrous. Presque aucune des fonctions de Node ne fait d'entrée/sortie directement, évitant les appels bloquant. Puisque rien n'est bloquant, des développeurs non-experts peuvent écrire des systèmes performants.

Node est similaire dans sa conception et est influencé par des systèmes tels que Event Machine pour Ruby ou Twistedpour Python. Node pousse le modèle évènementiel un peu plus loin en exposant la boucle d'évènement comme partie intégrante du langage et non comme une bibliothèque. Dans d'autres systèmes, il existe toujours un appel bloquant qui démarre la boucle d'évènement. Typiquement, les comportements et fonction de rappel doivent être définis au début du script avant de réaliser l'appel bloquant qui démarre le serveur tel que EventMachine::run(). Dans node il n'y a aucun appel pour démarrer la boucle. Node entre dans la boucle à la fin du script initial et en sort lorsque plus aucune fonction de rappel à éxécuter. Ce comportement ressemble à celui des navigateurs web, la boucle d'évènement est cachée à l'utilisateur.

HTTP est un protocole natif dans Node. La bibliothèque HTTP trouve son origine dans l'experience de son auteur et son travail avec les serveurs web. Par exemple, travailler avec des flux de données est impossible avec la plupart des frameworks. Node tente de corriger ces problèmes avec son parseur HTTP et son API. Couplée à l'infrastructure purement évènementielle de Node, cela offre de bonnes fondations aux bibliothèques ou frameworks web.

Mais qu'en est-il de la concurrence multi-processeurs ? N'a-t-on pas besoin de threads pour adapter les programmes aux ordinateurs multi-cœurs? Ce sont les processus qui sont nécéssaires à s'adapter aux ordinateurs multi-cœurs et non les threads à mémoire partagée. Les bases des systèmes scalables sont les réseaux rapides et une conception non-bloquante. Le reste est de la transmission de messages. Dans de futures versions, Node sera capable de fork()er de nouveaux processus en utilisant l'API Web Workers qui se marie bien au modèle actuel.
